Data Mash

  • Proposed by Toughtworks
  • Origin: https://martinfowler.com/articles/data-monolith-to-mesh.html

当前企业数据架构

特征：集中式、单体、领域不可知

不足

  • 第一代：企业数据仓库 + BI。代价昂贵，且存在巨大的技术债务，只有少部分人理解数据，导致在业务层面未能充分实现价值；
  • 第二代：数据湖。大数据生态过于复杂，长时间运行大量的 ETL 任务，且只有少部分专家能够熟悉并使用，期望值与实际收效不符；
  • 第三代：大致类似前两代，改进之处在于，1) 使用流计算实时处理数据，2) 流批一体，3) 拥抱云技术（存储、数据管道、机器学习）。第三代架构尝试解决数据实时处理问题，降低数据管理开销，但是很多特征注定会造成类似前两代的失败。

主要可以总结为，单体的灵活性和可扩展性较差，响应新需求（消费需求或者数据增加需求）都比较慢，每一次改动都牵涉较多

当前数据平台架构失败的原因

  • 集中式 + 单体
    • 功能职责
      • 注入数据：从各类数据源采集数据，并保存在数据平台；
      • 清洗、丰富和转换：根据需求处理数据；
      • 提供服务：满足分享、业务、智能等各类消费需求；
    • 问题：忽略了数据的领域划分，仅适合于领域简单、消费多样性低的企业；
      • 数据源众多导致数据多样化，该模式无法很好满足巨型企业多领域数据持续增加数据源，管理成本过高；
      • 数据平台的响应时间与快速创新数据消费需求的矛盾；
    • 澄清：
      • data mash 不是碎片化、孤立的组织面向领域的数据；
      • data mash 不是多个独立的积累了大量技术债的数据仓库；
      • 打破数据孤岛并非建立集中式数据平台一种途径，这存在规模瓶颈；
  • 耦合管道分解
    • 数据平台架构，其实就是对集成、清洗、聚合、服务等基本功能的解耦；
    • 主要任务：在上线新数据源和响应新消费需求不断增长下，架构师需要将系统化整为零，进行分解，有利于降低每个功能模块维护组织间的耦合性，提高工作的并行性；
    • 面对集中式数据平台，架构师需要将其分解为数据处理阶段的管道；
    • 优势：使系统在一定程度上具备可扩展性；
    • 问题：相干性减慢了发布特性的速率，前后两个阶段间紧耦合；
      • 某些平台提供了可配置数据源，但无法解决端到端的依赖管理；
      • 对于用户视角，还是需要引入新的数据集；
      • 虽然在看似数据管道的阶段间是松耦合的，实际上一整条管道，即整个平台才是最小的可变单位；
      • 架构分解和需求变更是正交的；
  • 孤立且高度专业化的所有权
    • 近距离观测构建和维护数据平台的团队，发现各个部分（数据引入、数据处理、数据消费）之间是完全隔离的，通常缺乏业务或领域的知识；
    • 缺乏对工程师提供高质量数据的激励；

Data Mesh

核心要素

  • Distributed Domain Driven 架构
  • Self-serve 平台设计
  • 技术数据的产品思考

数据和 DDD 结合

面向域的数据分解

  • 传统设计中，数据从领域流入集中仓储，而在 data mash 中领域维护自己的数据；
  • 数据的实际存储实现可以是集中的，但数据的产生、维护和消费是在相关领域内的；
  • 领域之间可能存在消费关系；
  • 强调在不同领域间可能存在数据重复，但这些数据在不同领域内会转换为最合适的形态；
  • 在面向域的数据平台中，基础组件是域，而非管道阶段；

面向源头的域数据

  • 源域数据集（source domain datasets）表示业务事实，即数据源位于业务系统中，源于反应某一项业务的运行情况；
  • 运行系统相关的业务团队，不进行需要实现业务能力，还需要提供能够反应系统真是运行情况的源域数据集；
  • 通常需要聚合多个源对齐数据集才能产生一个域对齐数据集；
  • 域事件（domain events）是业务事实的最佳表示；
  • 历史快照也是有必要的，它聚合了一段时间的事件；
  • 源域数据集非常基础，且很少改变，类似于原始数据；

面向消费者（多个消费者共享）的数据

  • 目标是满足相关组的用例；
  • 转换源域事件，聚合视图和结构来适配相应的数据访问模型；
  • 面向域的数据平台需要能够轻松的从原始数据生成不同的面向消费的数据集；

分布式管道

  • 虽然数据的所有权归于不同的域，但是数据仍然需要清洗、预处理、聚合、服务等步骤；
    • 清洗、标准化等阶段移入源域中；
    • 聚合等步骤移入消费域中；
  • 每个域数据集必须建立服务水平目标（service level objectives），保证时效性、错误率等；
  • 质疑：不同的域中存在重复实现数据管道阶段、工具等的情况；

数据和产品思维结合

在分布式的数据所有权和数据管道下，如何保证数据的可访问性（accessibility）、可使用性（usability）和协调（harmonization）。

域数据即产品

  • 域数据团队需要把数据作为自己的产品，其他数据科学家和机器学习、数据工程师都作为他们的客户；
  • 域数据的基本质量：
    • 可被发现（discoverable）：注册中心、数据目录、元数据管理（所有者、源头、血统、样本等）等技术实现；数据产品自我注册；
    • 可寻址（Addressable）：数据被发现后，需要提供唯一的地址方便用户访问；地址转换方式依赖存储类型和数据格式；一份数据可寻址的标准是关键；
    • 真实可信（trustworthy and truthful）：数据所有者需要围绕数据可信建立服务水平目标（service level objective，SLO）；评估数据反应事实的程度；数据清洗和完整性测试等技术常用来保证数据质量；不同域需要定义各自的SLOs;
    • 自我描述语义与语法（self-describing semantics and syntax）：可以以数据模式（schema）为起点，进行扩展；
    • 能够互操作且符合全局标准（inter-operable and governed by global standards）：关联不同领域的数据，整合为更理想的形式；制定符合全局数据治理规则的标准，包括字段类型、领域间词义消歧、数据地址转换、通用元数据字段、事件格式；
    • 全局访问控制（Secure and governed by a global access control）：每个域对自己的数据产品有细粒度的管理；实现方式和一般的分布式系统无异；
    • 跨职能团队（Domain data cross-functional teams）：两个新角色；data product owner 负责定义数据产品，对领域数据集的生命周期负责，应该定义一系列业务相关的关键指标；data engineer 负责建立并维护领域内的数据管道；技术授粉，类似于 DevOps；

数据与自服务平台设计的结合

  • 分布数据所有权带来最大的潜在问题：为了维护领域内数据管道基础设施造成冗余的数据、技术需求。可以通过构建通用的基础设施即服务（DIaaS）解决，但是相关技术在数据领域还不够成熟；
  • 由数据基础设施团队维护必要的技术，供各个领域用于采集、处理、存储和服务自己的数据；
  • 建设 Data Infrastructure as a Service（DIaaS）:
    • 不包括任何领域特有的概念和业务逻辑；
    • 屏蔽潜在的组件复杂性；
    • 面向领域数据工程师，需要具备：
      • Scalable polyglot big data storage，可扩展的多语言大数据存储；
      • Encryption for data at rest and in motion，静态和动态数据加密；
      • Data product versioning，数据产品版本；
      • Data product schema，数据产品模式（摘要）；
      • Data product de-identification，数据产品去标识（脱敏？）；
      • Unified data access control and logging，统一的数据访问控制和日志；
      • Data pipeline implementation and orchestration，数据管道实现和编排；
      • Data product discovery, catalog registration and publishing，数据产品发现、目录注册和发布；
      • Data governance and standardization，数据治理和标准化；
      • Data product lineage，数据产品血缘；
      • Data product monitoring/alerting/log，数据产品监控、告警和日志；
      • Data product quality metrics (collection and sharing)，数据产品质量指标（收集和分享）；
      • In memory data caching，数据缓存；
      • Federated identity management，联合身份管理；
      • Compute and data locality，计算和数据局部性；

总结

  • 当前数据平台的特征：集中、单体、数据管道紧耦合、职责割裂；
  • data mash：
    • 面向不同的领域的分布式数据产品，由独立的跨职能团队维护，使用统一的 DIaaS 服务；
    • 通过集中的治理标准实现互操作性；
  • data lake 等传统平台的工具不是被取代，而是角色转变，领域数据产品成为一等公民；
